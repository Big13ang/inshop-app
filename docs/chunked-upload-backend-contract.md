# Chunked Upload — Backend Contract

This document specifies the API the backend must implement to support the chunked upload client. The client handles resumability, parallel chunk concurrency, and SHA-256 integrity verification. The backend is responsible for storing chunks, verifying checksums, and assembling the final file.

---

## Overview

```
Client                                  Server
  │
  ├─ POST /chunk (up to 3 in parallel) ──► verify checksum → store chunk file
  │   (p-retry per chunk, max 3 retries)
  │
  ├─ GET /{fileId}/chunks ─────────────► return list of already-received indices
  │   (called only on resume)
  │
  ├─ POST /{fileId}/finalize?total=N ──► verify all N chunks present → assemble → return URL
  │◄─ { url } ──────────────────────────┤
```

### Key design decisions

| Decision | Reason |
|---|---|
| Client generates `fileId` (UUID v4) | Avoids a round-trip session initiation step |
| SHA-256 checksum per chunk | Catches bit corruption and partial transfers before writing to disk |
| Chunks uploaded in parallel (3 at a time) | ~3× throughput improvement for large video files |
| Explicit `/finalize` call | Removes the race condition of "detect last chunk by counting files" |
| `localStorage` resume state on client | Survives page refresh; client re-syncs with server via GET `/chunks` on restart |

## Endpoints

### 1. Upload a chunk

```
POST /api/upload/chunk
```

#### Query parameters

| Param | Type | Description |
|---|---|---|
| `fileId` | `string` (UUID) | Session identifier, generated by the client |
| `index` | `number` | Zero-based chunk index (0 … total-1) |
| `total` | `number` | Total number of chunks in this file |
| `filename` | `string` (URL-encoded) | Original filename, used for safe file naming on assembly |

#### Request headers

| Header | Required | Description |
|---|---|---|
| `Content-Type` | no | Can be omitted — body is raw bytes |
| `X-Chunk-Checksum` | **yes** | Lowercase hex SHA-256 of the raw chunk bytes |

#### Request body

Raw binary chunk bytes (up to `chunkSize`, default 5 MB).

#### Responses

| Status | Body | Meaning |
|---|---|---|
| `200` | `{ success: true, index: N }` | Chunk accepted and stored |
| `400` | `{ error: "Missing X-Chunk-Checksum header" }` | Header absent |
| `400` | `{ error: "Checksum mismatch" }` | SHA-256 of received bytes differs from header |
| `400` | `{ error: "Invalid index or total" }` | index ≥ total or not numbers |
| `429` | _(any)_ | Rate limited — client will retry with backoff |
| `5xx` | _(any)_ | Server error — client will retry with backoff |

#### Idempotency

The endpoint **must be idempotent**: uploading the same `(fileId, index)` a second time must succeed silently (overwrite the stored chunk). The client retries failed chunks; the server must not reject duplicates.

#### Checksum verification (pseudocode)

```
received_bytes = request.body
server_hash = sha256(received_bytes).hex()
client_hash = request.headers["X-Chunk-Checksum"]

if server_hash != client_hash:
    return 400 { error: "Checksum mismatch" }

write_to_disk(path = chunks/{fileId}/{index}, data = received_bytes)
```

#### Atomicity

Write chunks atomically to avoid partial reads during finalize:

```
write(path = chunks/{fileId}/{index}.tmp, data = bytes)
rename(chunks/{fileId}/{index}.tmp → chunks/{fileId}/{index})
```

### 2. Query uploaded chunks (resume)

```
GET /api/upload/{fileId}/chunks
```

Called by the client **only when resuming** a previously interrupted upload (detected via `localStorage`). Returns the indices the server already has so the client skips re-uploading them.

#### Path parameters

| Param | Type | Description |
|---|---|---|
| `fileId` | `string` (UUID) | Upload session to query |

#### Responses

| Status | Body | Meaning |
|---|---|---|
| `200` | `{ received: number[] }` | Indices of chunks stored on the server |
| `200` | `{ received: [] }` | Session not found — treat as fresh start |

Example response for a 10-chunk upload where chunks 0, 1, 3, 5 arrived:

```json
{ "received": [0, 1, 3, 5] }
```

The response does **not** need to be sorted. The client builds a set.

---

### 3. Finalize (assemble)

```
POST /api/upload/{fileId}/finalize?total=N
```

Called once by the client after all chunks have been successfully acknowledged. The server must verify all `N` chunks are present before assembling.

#### Query parameters

| Param | Type | Description |
|---|---|---|
| `total` | `number` | Expected total chunk count |

#### Path parameters

| Param | Type | Description |
|---|---|---|
| `fileId` | `string` (UUID) | Upload session to finalize |

#### Responses

| Status | Body | Meaning |
|---|---|---|
| `200` | `{ url: string }` | Assembly succeeded; `url` is the publicly accessible path |
| `404` | `{ error: "Upload session not found" }` | No chunks stored for this `fileId` |
| `409` | `{ error: "Missing chunks", missing: number[] }` | Not all chunks received yet |
| `5xx` | `{ error: string }` | Assembly failed |

The `url` value must be a path or full URL the client can use to display or submit the file:

```json
{ "url": "/media/uploads/abc123_video.mp4" }
```

#### Assembly requirements

1. Concatenate chunks in **strict ascending index order** (0, 1, 2, … N-1).
2. Verify `present_chunk_count === total` before starting — return `409` if mismatch.
3. Delete the temporary chunk directory after successful assembly.
4. The assembled file must be byte-for-byte identical to the original. No re-encoding.

---

## Error handling contract

The client uses exponential backoff (1 s → 2 s → 4 s, max 3 retries) for these status codes:

| Code | Retried? |
|---|---|
| `0` (network failure) | yes |
| `429` | yes |
| `500`, `502`, `503`, `504` | yes |
| `400` (bad checksum, missing header) | **no** — permanent failure |
| `404`, `409` on finalize | **no** — reported as upload failure |

## File size and limits

| Parameter | Default (client) | Notes |
|---|---|---|
| Chunk size | 5 MB | Configurable at `createChunkStrategy(fetch, chunkSize)` |
| Max parallel chunks | 3 per file | Inner concurrency; outer file concurrency is also 3 |
| Max retries per chunk | 3 | After 3 failures the whole file is marked failed |

## Storage recommendations

### Chunk storage

Store chunks in a temporary area keyed by `fileId`:

```
/tmp/uploads/{fileId}/0
/tmp/uploads/{fileId}/1
/tmp/uploads/{fileId}/2
...
```

Keep metadata (filename, total) alongside chunks:

```
/tmp/uploads/{fileId}/_meta.json  →  { "filename": "video.mp4", "total": 12 }
```

### Cleanup policy

Implement a background job to delete incomplete sessions older than **24 hours**. The client's `localStorage` key expires implicitly once the user uploads the same file again.

### Production storage

For production, replace local disk with:
- **Chunks**: object storage (S3, GCS, Cloudflare R2) — use `{fileId}/{index}` as the key
- **Metadata**: Redis (TTL = 24h) or a `upload_sessions` DB table
- **Assembly**: stream chunks directly into the final object; avoid loading all chunks into memory simultaneously

---

## Security notes

1. **Validate `fileId` format** — reject anything that is not a UUID v4 to prevent path traversal.
2. **Sanitize `filename`** — strip all characters except `[a-zA-Z0-9._-]` before using in file paths.
3. **Rate-limit the chunk endpoint** — the client sends up to 3 concurrent requests per file and retries up to 3 times, so a single upload can make up to `totalChunks × 3 × 3` requests in the worst case.
4. **Cap total upload size** — enforce a per-session maximum (e.g., 500 MB for video) to prevent storage abuse.
5. **Authenticate** — all endpoints should require a valid session/JWT. Pass it via the `Authorization` header; the current client does not set it but will need to once auth is wired up.

## Sequence diagram — fresh upload (10 chunks, 3 parallel)

```
Client                          Server
  │
  ├──[POST chunk 0]─────────────►[checksum OK, store chunk 0]
  ├──[POST chunk 1]─────────────►[checksum OK, store chunk 1]
  ├──[POST chunk 2]─────────────►[checksum OK, store chunk 2]
  │◄─ 200 ──────────────────────┤ (all three respond)
  │
  ├──[POST chunk 3]─────────────►[checksum MISMATCH → 400]
  │  retry → POST chunk 3 ──────►[checksum OK, store chunk 3]
  │
  ├──[POST chunk 4, 5, 6]───────►[store 4, 5, 6]
  │  ... continues ...
  │
  ├──[POST /finalize?total=10]──►[verify 10 chunks present → assemble]
  │◄─ { url: "/media/abc.mp4" }─┤
```
